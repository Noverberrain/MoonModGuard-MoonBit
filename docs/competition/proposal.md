# MoonModGuard 开源技术说明书

## 项目识别

**项目名称：** MoonModGuard：MoonBit 项目清单与供应链策略审计器

**GitHub 主仓库：** <https://github.com/Noverberrain/MoonModGuard-MoonBit->

**GitLink 镜像仓库：** <https://gitlink.org.cn/Wyc060514/moonmodguard>

**开源许可证：** Apache-2.0

## 评审摘要

MoonModGuard 面向 MoonBit 项目的工程清单质量与供应链策略检查，提供一个轻量、可测试、可嵌入的审计内核。它解析 `moon.mod` 与 `moon.pkg` 文本，抽取项目元数据、包级依赖、导入别名和发布信息，并将这些信息统一转化为可复查的策略诊断报告。

项目的首版目标不是实现完整语言解析器，也不是构建平台型服务，而是优先补齐 MoonBit 生态中较容易被忽略的“发布前自检”能力：项目能否说明身份、是否声明许可证、仓库与 README 是否完整、依赖来源是否符合本地策略、重复依赖是否会影响维护。

## 选题价值

MoonBit 生态会持续积累包、模板和教学工程。随着项目数量增加，单个仓库是否具备可发布、可维护、可审计的基础条件，会直接影响生态复用效率。现阶段这类检查通常依赖人工阅读清单文件，结果不稳定，也不容易接入 CI。

MoonModGuard 将上述检查沉淀为基础软件能力，适用于三类场景：包发布前的清单完整性检查、竞赛与课程仓库的工程规范验收、后续包仓库或 CI 系统中的自动化策略审计。

## 技术方案

系统采用四层结构。

1. 清单解析层：读取 `moon.mod` 与 `moon.pkg` 字符串，解析标量字段、数组字段和 import 块。
2. 项目建模层：合并模块级与包级信息，形成统一的 `ProjectModel`。
3. 策略评估层：依据 `Policy` 检查必填元数据、许可证 allowlist、可信依赖前缀和重复依赖。
4. 报告生成层：将 `AuditReport` 渲染为稳定 Markdown，供 CLI、CI 或文档系统复用。

首版刻意采用“显式输入字符串 + 内存快照”的方式，避免在文件系统扫描、网络访问和平台集成上引入额外不确定性。这使核心逻辑更容易测试，也便于后续扩展为真实工作区扫描器。

## 核心交付

- `parse_mod`：解析模块清单中的名称、版本、许可证、README、仓库地址、关键词、描述和依赖声明。
- `parse_pkg`：解析包级 import 依赖与别名。
- `project_from`：组合模块信息和包信息，构建项目模型。
- `evaluate_policy` / `scan_project`：执行策略检查并生成诊断报告。
- `render_markdown`：输出稳定、可提交、可比对的审计文本。
- `cmd/main`：提供可运行演示，展示依赖数量、风险数量和报告片段。

## 验证证据

当前仓库包含源码、测试、README、许可证、CI 配置和申报材料。验证命令包括：

```bash
moon info
moon fmt --check
moon test
moon run cmd/main
```

测试覆盖 `moon.mod` 标量字段解析、`keywords` 数组解析、模块级 import 解析、包级 import 解析、缺失元数据诊断、非 allowlist 许可证诊断、未知依赖前缀诊断、重复依赖诊断和 Markdown 报告稳定输出。

## 创新与边界

MoonModGuard 的创新点在于把 MoonBit 项目的工程元数据和供应链策略检查作为独立审计对象，而不是继续选择常见格式解析、通用数据结构、解释执行或内容寻址类方向。范围较窄，但具有明确工程价值：它关注包发布准备度、依赖策略透明度和清单质量可复查性。

首版边界也保持清晰：不做完整 MoonBit 语法解析，不做在线包仓库查询，不做平台服务绑定。后续可在稳定审计内核之上扩展真实文件扫描、工作区多包遍历、依赖图可视化、CI 阈值控制和包仓库审计。

## 开源计划

项目以 GitHub 仓库作为官方开源发布地址，满足赛事阶段一对有效 GitHub 仓库链接的要求；GitLink 仓库作为赛事平台镜像，用于同步提交和平台验收。后续迭代将优先保持核心 API 稳定、测试可复现、报告输出可比对，并逐步补充更多 MoonBit 工程场景。
